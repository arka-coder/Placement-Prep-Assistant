DBMS (Database Management System) Study Notes

1. NORMALIZATION
Definition: Normalization is the process of organizing a relational database schema to minimize data redundancy, eliminate data anomalies (insertion, update, and deletion anomalies), and ensure data integrity. It works by decomposing large tables into smaller, well-structured tables and defining relationships between them using foreign keys.
Explanation: Normalization is guided by a series of rules called Normal Forms (NF). Each higher normal form builds on the previous one by removing specific types of data dependencies. The goal is to ensure that each piece of information is stored in exactly one place — following the DRY (Don't Repeat Yourself) principle. Without normalization, updating one fact in a database may require updating multiple rows, leading to inconsistencies.
Interview Tip: Most production databases are normalized to 3NF. BCNF is stricter but can sometimes cause lossless decomposition to be impossible. In data warehouses, denormalization is deliberately applied for faster read performance (star schema, snowflake schema).

First Normal Form (1NF): Each column must contain atomic (indivisible) values — no multi-valued attributes, no repeating groups. Every row must be uniquely identifiable. Example violation: Storing multiple phone numbers in a single "Phone" column. Fix: Create separate rows for each phone number.

Second Normal Form (2NF): Must be in 1NF AND every non-key attribute must be fully functionally dependent on the entire primary key (no partial dependency). This is relevant only when the primary key is composite (multi-column). Example violation: In (StudentID, CourseID) → StudentName — StudentName depends only on StudentID, not the full composite key. Fix: Move StudentName to a separate Students table.

Third Normal Form (3NF): Must be in 2NF AND no non-key attribute should transitively depend on the primary key through another non-key attribute. Example violation: EmployeeID → DepartmentID → DepartmentName — DepartmentName depends on DepartmentID, not directly on EmployeeID. Fix: Create a separate Departments table.

Boyce-Codd Normal Form (BCNF): Stricter than 3NF. For every functional dependency X → Y, X must be a superkey (candidate key). Eliminates anomalies that 3NF may leave behind when there are multiple overlapping candidate keys.

2. INDEXING
Definition: An Index is a database object that improves the speed of data retrieval operations on a table at the cost of additional storage and slower write operations. It is analogous to the index at the back of a book — you look up the topic and go directly to the right page instead of reading the entire book.
Explanation: Without an index, a database performs a Full Table Scan — reading every row to find matches (O(n)). With an index, retrieval becomes O(log n) using B-Tree indexes or O(1) for hash indexes. Types of indexes: Primary Index (automatically created on the primary key, ensures uniqueness), Secondary Index (on non-primary key columns to speed up queries), Unique Index (enforces uniqueness on a column), Composite Index (index on multiple columns — efficient for queries filtering on those columns in order), and Full-Text Index (for searching within large text fields). Trade-off: Every index speeds up SELECT queries but slows down INSERT, UPDATE, and DELETE operations because the index must be maintained.
B-Tree Index: The most common type. A self-balancing tree structure where all leaf nodes are at the same depth. Search, insert, and delete all take O(log n). Excellent for both exact match and range queries (BETWEEN, <, >). Used by default in MySQL (InnoDB), PostgreSQL, and most relational databases.
Hash Index: Uses a hash function to map keys to array positions. Provides O(1) average-case lookup for exact match queries. However, it does NOT support range queries (cannot use BETWEEN or ORDER BY). Used in MySQL MEMORY tables and some NoSQL stores.
Interview Tip: Composite indexes follow the "left-most prefix rule" — an index on (A, B, C) can speed up queries filtering on A, or A+B, or A+B+C, but NOT on B alone or C alone. Always consider this when designing composite indexes.

3. ACID PROPERTIES
Definition: ACID stands for Atomicity, Consistency, Isolation, and Durability — the four fundamental properties that guarantee reliable processing of database transactions, even in the presence of errors, power failures, and other failures.
Atomicity: A transaction is treated as a single, indivisible unit. It either completes fully (all operations succeed and are committed) or not at all (if any operation fails, the entire transaction is rolled back). Example: In a bank transfer, both debit and credit must succeed together — if the debit succeeds but credit fails, the debit must be rolled back.
Consistency: A transaction brings the database from one valid state to another valid state, never leaving it in an inconsistent or intermediate state. All database rules (constraints, cascades, triggers) are enforced before and after a transaction. Example: A "funds transfer" transaction must not violate the constraint that bank account balances cannot be negative.
Isolation: Concurrent transactions are executed as if they were serial (one at a time). Each transaction's intermediate state is invisible to other concurrent transactions. Isolation levels (Read Uncommitted, Read Committed, Repeatable Read, Serializable) provide configurable trade-offs between isolation strength and concurrency performance.
Durability: Once a transaction is committed, its changes are permanent — they survive system crashes, power failures, or hardware errors. This is implemented through write-ahead logging (WAL) — changes are written to a durable log before being applied to the database.
Interview Tip: Not all databases guarantee full ACID compliance. Traditional SQL databases (MySQL InnoDB, PostgreSQL) are ACID-compliant. Some NoSQL databases (e.g., Cassandra, MongoDB) sacrifice full ACID for scalability — they follow BASE (Basically Available, Soft state, Eventually consistent).

4. TRANSACTIONS AND CONCURRENCY CONTROL
Definition: A Transaction is a sequence of one or more database operations (read, write, update, delete) that is treated as a single logical unit of work. Transactions must satisfy ACID properties to ensure data integrity in multi-user database environments.
Explanation: Transaction States: Active (currently executing), Partially Committed (all operations done, not yet confirmed), Committed (successfully completed and changes saved permanently), Failed (error occurred), Aborted (rolled back, database restored to state before the transaction began). Concurrency problems when transactions execute simultaneously: Dirty Read (reading uncommitted data from another transaction), Non-Repeatable Read (same row returns different values when read twice in same transaction), Phantom Read (new rows appear in a range query during a transaction because another transaction inserted them). Concurrency Control mechanisms: Locking (Shared/Read Lock — multiple transactions can read; Exclusive/Write Lock — only one transaction can write), Two-Phase Locking (2PL) — ensures serializability, Timestamp Ordering, Multiversion Concurrency Control (MVCC — used by PostgreSQL, allows readers and writers not to block each other).
Deadlock: Occurs when two or more transactions are each waiting for the other to release a lock — circular wait. Prevention strategies: Wait-Die scheme, Wound-Wait scheme, lock timeout. Detection: Maintain a wait-for graph and detect cycles.
Interview Tip: SQL isolation levels map directly to which concurrency anomalies they prevent. SERIALIZABLE is the strongest (prevents all anomalies) but lowest concurrency. READ COMMITTED is the PostgreSQL default — prevents dirty reads but allows non-repeatable reads. Know the trade-offs.

5. SQL — KEY CONCEPTS AND JOINS
Definition: SQL (Structured Query Language) is the standard language for managing and querying relational databases. It is used to define schema (DDL), manipulate data (DML), control access (DCL), and manage transactions (TCL).
Explanation of Joins: INNER JOIN returns only rows where there is a matching value in both tables. LEFT (OUTER) JOIN returns all rows from the left table and matched rows from the right table — unmatched right rows are NULL. RIGHT (OUTER) JOIN is the mirror of LEFT JOIN. FULL OUTER JOIN returns all rows from both tables — unmatched rows on either side are NULL. CROSS JOIN returns the Cartesian product of both tables (all combinations). SELF JOIN joins a table with itself — used for hierarchical queries (e.g., finding employees and their managers in the same table).
Key SQL Clauses: GROUP BY groups rows that share values in specified columns for aggregate functions. HAVING filters groups after aggregation (unlike WHERE, which filters before aggregation). ORDER BY sorts results. DISTINCT removes duplicate rows. LIMIT/OFFSET for pagination.
Aggregate Functions: COUNT(), SUM(), AVG(), MIN(), MAX() — always used with GROUP BY or on entire table.
Interview Tip: Common SQL interview question pattern: "Find the second highest salary" → SELECT MAX(salary) FROM employees WHERE salary < (SELECT MAX(salary) FROM employees). Also know the difference between WHERE and HAVING — WHERE filters rows before grouping; HAVING filters groups after grouping.

6. ENTITY-RELATIONSHIP (ER) MODEL
Definition: The Entity-Relationship (ER) Model is a high-level conceptual data model used to describe the logical structure of a database. It represents data as entities (objects), their attributes (properties), and the relationships between them, typically visualized as an ER Diagram (ERD).
Explanation: An Entity is a real-world object (e.g., Student, Course, Employee). An Attribute is a property of an entity (e.g., StudentID, Name, Age). Key types: Primary Key (unique identifier), Composite Attribute (made up of sub-attributes), Multi-Valued Attribute (can have multiple values), Derived Attribute (computed from other attributes). A Relationship describes how entities are associated. Cardinality specifies the numerical relationships: One-to-One (1:1), One-to-Many (1:N), Many-to-Many (M:N). Many-to-Many relationships are implemented with a junction table. Weak Entity depends on a strong entity for its existence (e.g., a Dependent entity depends on an Employee entity).
Interview Tip: The ER diagram is always the first step in database design. When converting an ER diagram to relational tables: each entity becomes a table, each attribute becomes a column, and each relationship is mapped using foreign keys (or a junction table for M:N relationships).

7. QUERY OPTIMIZATION
Definition: Query Optimization is the process by which a database management system (DBMS) automatically selects the most efficient execution plan for a SQL query from among potentially many equivalent plans. The goal is to minimize resource consumption (CPU, I/O, memory) and response time.
Explanation: The DBMS's query optimizer generates multiple equivalent execution plans and estimates their cost using database statistics (table sizes, index selectivity, column value distributions). The plan with the lowest estimated cost is chosen. Key optimization strategies: Use indexes to avoid full table scans, push predicates (WHERE conditions) as early as possible (predicate pushdown), choose the optimal join order (joining smaller tables first reduces intermediate result sizes), use query hints to override optimizer decisions when necessary. The EXPLAIN or EXPLAIN ANALYZE command (in PostgreSQL/MySQL) shows the query execution plan — essential for identifying bottlenecks.
Interview Tip: Always add indexes on foreign keys and columns frequently used in WHERE, JOIN ON, ORDER BY, and GROUP BY clauses. Avoid using functions on indexed columns in WHERE clauses (e.g., WHERE YEAR(created_at) = 2024) — this prevents index usage. Instead, use range conditions: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'.

8. KEYS IN DATABASES
Definition: Keys in a relational database are attributes or sets of attributes used to uniquely identify rows in a table and establish relationships between tables. Different types of keys serve different purposes in schema design and integrity enforcement.
Explanation: Primary Key (PK): Uniquely identifies each row in a table. Cannot be NULL. Only one primary key per table. Example: StudentID in a Students table. Foreign Key (FK): An attribute in one table that references the Primary Key in another table. Enforces referential integrity — prevents orphan records. Example: CourseID in an Enrollments table referencing CourseID in a Courses table. Candidate Key: A minimal set of attributes that can uniquely identify a row. There can be multiple candidate keys; one is chosen as the Primary Key, and the rest are called Alternate Keys. Super Key: Any set of attributes that can uniquely identify a row (may include non-minimal sets). Every candidate key is a super key, but not every super key is a candidate key. Composite Key: A primary key composed of two or more attributes. Used when no single attribute is unique by itself. Surrogate Key: An artificial primary key (often an auto-incremented integer or UUID) created by the database system when no natural key exists.
Interview Tip: Always prefer a surrogate key (auto-incremented ID) over a natural key (like email or phone number) for performance and stability — natural keys can change, but surrogate keys never do.
